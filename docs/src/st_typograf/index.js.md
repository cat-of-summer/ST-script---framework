## 1. Описание

`st_typograf` — статичный класс-утилита для микротипографики русского текста.
Его единственный публичный метод `preventWidows(entity)` устраняет «висячие»
слова: предлоги, союзы и частицы, которые иначе оставались бы в конце строки
перед переносом, склеиваются со следующим словом неразрывным пробелом
(` `, `&nbsp;`).

**Ключевые принципы:**

- **Никакой обёртки тегами.** Склейка пишется прямо в `nodeValue` текстовой
  ноды (символ NBSP вместо обычного пробела). DOM-структура и количество
  текстовых нод не меняются — это важно для совместимости с другими
  библиотеками, которые их обходят.
- **Идемпотентность.** Корневой элемент и все его потомки помечаются приватным
  `Symbol`-маркером. Повторный вызов `preventWidows` на той же сущности — no-op.
- **Реактивность.** На корневом элементе сразу же активируются три
  наблюдателя — `MutationObserver`, `IntersectionObserver`, `ResizeObserver` —
  чтобы текст оставался аккуратным при изменениях DOM, ширины контейнера и
  появлении в области видимости.
- **Пересчёт из оригинала.** Для каждой обработанной текстовой ноды
  сохраняется исходное значение в `WeakMap`. При ресайзе склейка строится
  заново именно от него — NBSP не накапливаются.

## 2. Что делает `preventWidows`

`preventWidows(entity)` выполняет следующее по шагам:

1. **Резолв сущности.** `entity` — это `Element` или CSS-селектор. Внутренний
   `#find_element` пропускает `Element` без изменений или ищет первый
   совпавший узел через `document.querySelector`. Если ничего не найдено —
   тихий выход.
2. **Защита от повторного применения.** Если на корне уже стоит
   Symbol-маркер — выход.
3. **Маркировка поддерева.** На корневом элементе и всех его дочерних
   `Element`-узлах ставится Symbol-маркер.
4. **Сбор текстовых нод.** `TreeWalker` обходит поддерево с фильтром
   `NodeFilter.SHOW_TEXT`, отсеивая пустые/пробельные ноды и нод-потомков
   запрещённых тегов / `contenteditable`.
5. **Сохранение оригиналов.** Для каждой подходящей текстовой ноды её
   `nodeValue` записывается в `WeakMap`, узел помечается тем же
   Symbol-маркером.
6. **Применение типографики.** Для каждой ноды вызывается `#apply` —
   `nodeValue` перезаписывается результатом `#typograf(original)`.
7. **Подключение наблюдателей.** На корневом элементе создаются и стартуют
   `MutationObserver`, `IntersectionObserver` и `ResizeObserver`, их состояние
   хранится в общем `WeakMap`, ключом которого является корневой элемент.

## 3. Словарь «висячих» слов

Слова заданы статичным приватным `Set` (`#SHORT_WORDS`) и охватывают три
основных класса служебных слов русского языка.

| Класс | Слова |
|---|---|
| **Предлоги** | в, во, на, по, с, со, к, ко, у, о, об, обо, за, из, изо, до, от, при, под, подо, над, надо, без, для, про, через, сквозь, между, меж, перед, передо, около, среди, кроме, вокруг, вдоль, против, ради, вместо, внутри, сверху, снизу, возле, напротив, благодаря, согласно, вопреки |
| **Союзы** | и, а, но, да, или, либо, что, чтоб, чтобы, как, если, ибо, хотя, пока, когда, куда, откуда, зачем, почему |
| **Частицы** | же, ли, бы, б, ведь, лишь, уж, аж, то, ни, не, ну, вон, вот |

### Регулярное выражение

Из словаря собирается одно регулярное выражение `#word_regex`:

```js
new RegExp(`(?<=^|[\\s(«„"'\\[])(${escaped})(\\s+)(?=\\S)`, 'gi')
```

| Часть | Смысл |
|---|---|
| `(?<=^|[\s(«„"'\[])` | Lookbehind: перед коротким словом — начало строки, пробел или открывающая пунктуация. Не «потребляет» границу, чтобы корректно обрабатывались подряд идущие служебные слова. |
| `(${escaped})` | Альтернация по словам из `#SHORT_WORDS`, отсортированным **по длине по убыванию**, чтобы `во` матчилось раньше `в`. |
| `(\s+)` | Сам пробельный разделитель — заменяется на NBSP. Берётся `\s+`, чтобы корректно работать с переводами строк и табуляциями внутри текстовой ноды. |
| `(?=\S)` | Lookahead: после пробелов должно быть не-пробельное слово (иначе склеивать не к чему). |
| `gi` | Глобальный + регистронезависимый. |

В замене используется только первая группа (само слово) + литерал ` `:

```js
text.replace(this.#word_regex, (m, word) => word + ' ');
```

Это сохраняет исходный регистр слова без вмешательства.

## 4. Дополнительные правила склейки

Помимо служебных слов, `#typograf(text)` применяет ещё два правила.

### Цифра + следующее слово

```js
/(\d) +(?=\S)/g  →  '$1 '
```

После цифры и перед любым непробельным словом обычный пробел заменяется на
NBSP. Это охватывает классические случаи:

| Исходный текст | Результат |
|---|---|
| `5 кг` | `5 кг` |
| `100 000 рублей` | `100 000 рублей` |
| `Версия 2 уже вышла` | `Версия 2 уже вышла` (плюс `2` склеена со словом) |

### Пробел перед тире

```js
/ +(?=[—–] )/g  →  ' '
```

Один или несколько обычных пробелов **перед** длинным (`—`) или средним (`–`)
тире, окружённым пробелом справа, заменяются на NBSP. Тире остаётся
прикреплённым к предыдущему слову и не «прыгает» на следующую строку.

| Исходный текст | Результат |
|---|---|
| `Москва — столица` | `Москва — столица` |
| `Длина — 5 метров` | `Длина — 5 метров` |

## 5. Публичный API

Класс предоставляет один статичный метод. Конструктор не используется.

### `st_typograf.preventWidows(entity)`

| Параметр | Тип | Описание |
|---|---|---|
| `entity` | `Element \| string` | DOM-элемент или CSS-селектор. Если строка — берётся первый совпавший узел через `document.querySelector`. |

**Возвращает:** ничего (`undefined`).

**Поведение:**

- Если `entity` не резолвится в `Element` — тихий выход.
- Если корневой элемент уже обработан (есть Symbol-маркер) — тихий выход.
- Иначе — выполняются шаги 3–7 из раздела [2](#2-что-делает-preventwidows).

```js
// Element
st_typograf.preventWidows(document.querySelector('.article'));

// CSS-селектор
st_typograf.preventWidows('#content');

// Документ целиком
st_typograf.preventWidows(document.body);
```

## 6. Защита от повторного применения

Используется статичный приватный `Symbol`:

```js
static #marker = Symbol();
```

Маркер ставится:

- на **корневой элемент** при первом вызове;
- на **каждый дочерний `Element`** в момент маркировки поддерева;
- на **каждую обработанную текстовую ноду** (`Text`-нода тоже может хранить
  произвольные свойства через JS).

Перед началом обработки `preventWidows` проверяет
`if (root[#marker]) return;`. Это означает:

| Сценарий | Поведение |
|---|---|
| `preventWidows('#a')` дважды | Второй вызов — no-op. |
| `preventWidows('#a')`, затем `preventWidows('#a-child')` | Второй вызов — no-op, потому что `#a-child` уже помечен как дочерний `#a`. Это сознательно: внутри уже-обработанного дерева не должно появляться второй комплект наблюдателей. |
| `preventWidows('#a')`, затем `preventWidows('#b')` (несвязанные) | Оба обрабатываются независимо, у каждого свой набор наблюдателей. |

> **Symbol — приватный.** Извне его прочитать нельзя. Если нужна диагностика,
> добавьте флаг через CSS-атрибут или временную глобальную переменную.

## 7. Наблюдатели

После первичной обработки на корневой элемент вешаются три наблюдателя. Их
общее состояние и ссылки на сами `*Observer` хранятся в статичном
`WeakMap` (`#observers`), ключом которого является корневой элемент:

```js
{
    mutation, intersection, resize,    // сами Observer-ы
    mutating,                          // guard: мы сейчас сами правим DOM
    visible,                           // элемент пересекается с viewport
    pending_resize,                    // ресайз пришёл, пока было невидимо
    flush_scheduled,                   // уже запланирован requestIdleCallback
    queue,                             // Set<TextNode> от MutationObserver
}
```

### MutationObserver — ленивый

Конфиг: `{ subtree: true, childList: true, characterData: true }`.

При каждом батче записей обработчик:

1. **Guard.** Если `state.mutating === true` — выход (это были наши собственные
   правки `nodeValue`).
2. Перебирает записи:
   - **`characterData`** — добавляет `r.target` в очередь (`state.queue`).
   - **`childList`** — для каждой `addedNodes`:
     - Если это текстовая нода — добавляет её в очередь.
     - Если это `Element` — помечает её и всё её поддерево Symbol-маркером,
       прогоняет `#collect_text_nodes` и складывает найденные текстовые ноды
       в очередь.
3. Если `state.visible === true` и очередь непуста — планирует флаш через
   `#flush_queue`.

**Ленивость** достигается двумя механизмами:

- **`requestIdleCallback`** (с фоллбэком `setTimeout(…, 0)`) — флаш
  выполняется не на той же микрозадаче, а в простое браузера.
- **Гейтинг по видимости.** Пока `state.visible === false`, очередь только
  пополняется — никакие правки не пишутся в DOM.

### IntersectionObserver

Опции: `{ rootMargin: '200px' }` — обработка стартует за 200px до фактического
появления в viewport, чтобы пользователь не успел увидеть «дёрганье» текста.

Наблюдает **только сам корневой элемент** (не каждую текстовую ноду — это было
бы избыточно).

При смене состояния:

| `isIntersecting` | Действие |
|---|---|
| `true` | `state.visible = true`. Если есть отложенная очередь — сразу `#flush_queue`. Если выставлен `state.pending_resize` — выполняет `#reapply_all` и сбрасывает флаг. |
| `false` | `state.visible = false`. Дальнейшие мутации и ресайзы накапливаются. |

### ResizeObserver

Наблюдает сам корневой элемент. При каждом срабатывании:

1. Если `state.mutating === true` — выход (мы сами могли спровоцировать
   ресайз правкой текста).
2. Если `state.visible === false` — выставляется `state.pending_resize = true`
   и выход. Реальный пересчёт случится при появлении в viewport.
3. Иначе — выполняется `#reapply_all(root)`: все известные текстовые ноды под
   корнем восстанавливаются из оригинала в `WeakMap` и переразбираются. Это
   гарантирует, что NBSP не накапливаются — каждый ресайз даёт результат как
   «с чистого листа».

### Guard `state.mutating`

Любая запись в `nodeValue` оборачивается в `#mutating_run(state, fn)`:

```js
state.mutating = true;
try { fn(); }
finally {
    state.mutation && state.mutation.takeRecords();
    state.mutating = false;
}
```

- `state.mutating = true` подавляет реакцию `MutationObserver`/`ResizeObserver`
  на наши собственные правки.
- `mutation.takeRecords()` сливает накопленные записи **до** возвращения флага
  в `false`, чтобы они не разрядились следующей микрозадачей и не запустили
  бесконечный цикл.

## 8. Пропускаемые теги и contenteditable

`#collect_text_nodes(root)` отбрасывает текстовые ноды, у которых среди
предков (вплоть до самого `root` включительно) встречается:

- любой тег из `#SKIP_TAGS`:
  `SCRIPT`, `STYLE`, `NOSCRIPT`, `CODE`, `PRE`, `TEXTAREA`, `KBD`, `SAMP`;
- любой элемент с `contenteditable=""` или `contenteditable="true"`.

Пустые и состоящие только из пробелов ноды также не обрабатываются.

Поведение применимо как к первичному обходу, так и к нодам, пришедшим через
`MutationObserver` (он использует ту же утилиту для добавленных элементов).

## 9. Примеры

### Базовое применение

```html
<article id="post">
    <p>В небольшом доме на окраине города жил старый часовщик.</p>
    <p>Он умел делать такие часы, что они шли точно.</p>
</article>

<script src="st_typograf/st_typograf.js"></script>
<script>
    st_typograf.preventWidows('#post');
</script>
```

После применения — внутри `<p>` останутся обычные пробелы между основными
словами, а после `в`, `на`, `что` и так далее будет ` `.

### Применение ко всей странице

```js
document.addEventListener('DOMContentLoaded', () => {
    st_typograf.preventWidows(document.body);
});
```

> Под одним общим корнем будет один комплект наблюдателей. Это самый
> экономный вариант: вместо N узлов отслеживается один.

### Динамический контент

```js
st_typograf.preventWidows('#feed');

// Где-то позднее — добавили карточку через fetch
const card = document.createElement('article');
card.innerHTML = `<p>Новая запись с предлогом и союзом.</p>`;
feed_root.appendChild(card);

// MutationObserver увидит childList-мутацию, прогонит #collect_text_nodes,
// положит ноду в очередь и обработает её в следующем requestIdleCallback.
```

Никаких ручных вызовов после вставки нового DOM не требуется.

### Адаптивная ширина

```html
<div id="card" style="resize: horizontal; overflow: auto; width: 480px;">
    <p>Текст, который должен корректно перевёрстываться при ресайзе.</p>
</div>
```

```js
st_typograf.preventWidows('#card');
```

При перетаскивании ручки ресайза `ResizeObserver` сработает, оригинал
извлечётся из `WeakMap`, типограф будет применён заново — NBSP не накопятся.

### Идемпотентность

```js
st_typograf.preventWidows('#a');
st_typograf.preventWidows('#a');           // no-op
st_typograf.preventWidows(document.querySelector('#a'));   // no-op
```

## 10. Известные особенности и ограничения

### Только русский язык

Словарь `#SHORT_WORDS` содержит исключительно русские служебные слова.
Английские артикли и предлоги (`a`, `an`, `the`, `of`, `in`, …) не обрабатываются.

### Склейка не пересекает границы текстовых нод

Регулярные выражения работают в пределах одной `Text`-ноды. Если соседние
текстовые ноды разделены инлайн-тегом (`<span>`, `<em>`), висячее слово в
конце одной ноды не будет склеено со словом из соседней. На практике это
редкая ситуация — в обычной разметке такие границы как правило соответствуют
смысловым переходам.

### Контент в `contenteditable` не трогается

Если поддерево объявлено как редактируемое, типограф пропустит его целиком —
иначе любая правка пользователя породила бы цикл MO → правка → MO.

### Lookbehind в регулярном выражении

`#word_regex` использует lookbehind (`(?<=…)`). Это требует браузер уровня
**Safari 16.4+** (март 2023), современные Chrome/Firefox/Edge поддерживают
давно.

### MutationObserver реагирует на ВСЁ поддерево

`{ subtree: true }` означает, что наблюдатель сработает на любую мутацию
любого потомка. Если в обработанное поддерево вставляется большой и
высоко-активный компонент (например, виртуализированный список), будьте
готовы к тому, что очередь будет наполняться чаще обычного. `requestIdleCallback`
смягчает нагрузку, но не устраняет её полностью.

### Невозможно «снять» применение

Публичного метода `release` нет. Если нужно отключить наблюдатели и
восстановить оригинальный текст — это придётся сделать вручную (отсоединить
поддерево от DOM или удалить корневой узел; `WeakMap`-записи освободятся
сборщиком мусора).

### `auto_close` / другие модули, перемещающие DOM

Если корневой элемент перемещают в другую часть DOM (например, `st_modal`
переносит контент в свой `<container>`), то наблюдатели **продолжат работать**
(они привязаны к самому узлу, а не к точке вставки). Symbol-маркер тоже
остаётся, поэтому повторный `preventWidows` после перемещения по-прежнему
будет no-op.

### Производительность на больших документах

`TreeWalker` по тысячам нод — операция O(N). Для очень больших страниц
(десятки тысяч узлов) разумнее вызывать `preventWidows` точечно по разделам,
а не на `document.body`.
